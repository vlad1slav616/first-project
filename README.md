
# Тема 1: Введение в Git  

## Что такое Git?  

Git — это распределённая система контроля версий, которая позволяет отслеживать изменения в файлах и координировать работу над проектами между несколькими разработчиками.

## Зачем нужен Git?

* Для отслеживания истории изменений в проекте.
* Для возможности возврата к предыдущим версиям кода.
* Для совместной работы над проектом командой разработчиков.

## Основные понятия:

* Репозиторий (repository) — место, где хранятся все файлы проекта и история их изменений.
* Коммит (commit) — запись об изменениях, сделанных в проекте.
* Ветвь (branch) — независимая линия разработки, которая может быть объединена с другими ветками.
* Индекс (staging area) — промежуточная область, где собираются изменения перед коммитом.

# Тема 2: Работа с Git

## Основные команды Git:

* git init — инициализация нового репозитория Git.
* git status — проверка состояния репозитория (какие файлы изменены, добавлены или удалены).
* git add — добавление файлов в индекс (стадию подготовки к коммиту).
* git commit — сохранение изменений в истории репозитория с комментарием.
* git push — отправка изменений на удалённый сервер.

## Создание репозитория:

* Создание локального репозитория командой git init.
* Добавление файлов в репозиторий и их отслеживание.
* Работа с удалённым репозиторием:
* Связывание локального репозитория с удалённым сервером.
* Отправка изменений на удалённый сервер командой git push.

## Хеш  
Что такое хеш. Хеширование коммитов
Хеширование (от англ. hash, «рубить», «крошить», «мешанина») — это способ преобразовать набор данных и получить их «отпечаток» (англ. fingerprint).
Информация о коммите — это набор данных: когда был сделан коммит, содержимое файлов в репозитории на момент коммита и ссылка на предыдущий, или родительский (англ. parent), коммит.
Git хеширует (преобразует) информацию о коммите с помощью алгоритма SHA-1 (от англ. Secure Hash Algorithm — «безопасный алгоритм хеширования») и получает для каждого коммита свой уникальный хеш — результат хеширования.

## Лог
Лог (Log) - История коммитов в репозитории. - Позволяет отслеживать изменения, просматривать авторов и даты коммитов. - Команда git log отображает историю коммитов с деталями.
Получить сокращённый лог можно с помощью команды git log с флагом --oneline (англ. «одной строкой»). В терминале появятся только первые несколько символов хеша каждого коммита и их комментарии.
Сокращённый лог полезен, если в репозитории уже много коммитов — например, сотни или тысячи. В этом случае можно быстро найти нужный по описанию.

## Файл HEAD
Файл HEAD (англ. «голова», «головной») — один из служебных файлов папки .git. Он указывает на коммит, который сделан последним (то есть на самый новый).
Внутри HEAD — ссылка на служебный файл: refs/heads/master (или refs/heads/main в зависимости от названия ветки). Если заглянуть в этот файл, можно увидеть хеш последнего коммита.
При работе с Git указатель HEAD используется довольно часто. Мы уже упоминали, что многие команды Git принимают в качестве параметра хеш коммита.

## Статусы файлов в Git

### Основные статусы файлов в Git

1.Untracked (Неотслеживаемый): файл, который не отслеживается Git и не был добавлен в индекс. Это новый файл в вашем проекте, который ещё не добавлен в репозиторий. 
Пример: вы создали новый файл new_file.py, но ещё не добавили его в Git.

2.Tracked (Отслеживаемый): файлы, которые Git отслеживает и знает о них. Они могут иметь несколько подстатусов: 
 
2.1 Committed (Зафиксированный): файл был добавлен в последний коммит.  
Пример: после выполнения git commit -m "Initial commit", все добавленные файлы переходят в этот статус.  
2.2 Staged (Подготовленный): изменения в файле были добавлены в индекс (с использованием git add) и готовы к коммиту.  
Пример: git add modified_file.py подготавливает файл к следующему коммиту.  
2.3 Modified (Изменённый): файл был изменён, но изменения ещё не добавлены в индекс.  
Пример: если вы внесли изменения в modified_file.py, Git будет считать его изменённым до тех пор, пока вы не добавите изменения в индекс.  
2.4 Deleted (Удалённый): файл удалён из рабочего каталога и готов к фиксации.  
Пример: использование git rm removed_file.py помечает файл как удалённый и добавляет это изменение в индекс.

## Жизненный цикл файлов в Git

Жизненный цикл файла в Git можно представить следующим образом:  
#####Создание файла (Untracked):
  
Вы создаёте новый файл в вашем проекте.  

Git не знает о его существовании, пока вы не добавите его в репозиторий.

##### Добавление файла в отслеживание (Tracked):

Выполняете git add new_file.py, чтобы добавить файл в индекс.

Теперь Git отслеживает файл и готов зафиксировать его изменения.

##### Фиксация изменений (Committed):

После git commit -m "Add new file", изменения фиксируются в истории репозитория.

Файл становится зафиксированным в текущем состоянии.

##### Изменение файла (Modified):
Когда вы вносите изменения в файл, Git отмечает его как изменённый.

Чтобы зафиксировать изменения, снова нужно добавить файл в индекс с помощью git add modified_file.py.
Удаление файла (Deleted):

Если вы удаляете файл с помощью git rm removed_file.py, Git помечает его как удалённый.

Фиксация этих изменений завершает удаление файла из репозитория.

### Пример работы с файлами

1.Создаём новый файл new_file.py:

```mermaid 
 touch new_file.py
```
2.Добавляем файл в отслеживание:
```mermaid
   git add new_file.py
```
3.Фиксируем изменения:
```mermaid 
    git commit -m "Initial commit"
```
4.Вносим изменения в new_file.py и добавляем их в индекс:
```mermaid 
  nano new_file.py # вносим изменения
   git add new_file.py
   git commit -m "Update new file"
```
Удаляем файл:
```mermaid
bash git rm new
```
